Spec-Driven Development, Minus the Vibes: A Scientific Approach to Coding Agents

Why my coding agent can write the code, tests, plan, and review notes, but still does not get the final merge.

There is a tempting trap with stronger models: give them more freedom.

In coding, that goes wrong fast. The agent writes a lot. It sounds confident. Sometimes the code even looks clean. Then you open the diff and realize that along with the useful work, it made decisions you would never have approved yourself.

After a few rounds of that, I ended up with a simple boundary.

The AI can propose. I do the merge.

Not because I do not trust models. I use them every day, and pretty aggressively. The problem is simpler: a coding agent does not own the consequences of its decisions. It can write code quickly, but it does not know where "looks good" stops being good enough.

A human can catch that. But only if the process has an actual human decision point, not just a final confident summary from the model.

That is why my work on map-framework is not really about making the model smarter. I am not improving Claude or Codex. I am improving the environment around them, so their output can survive a serious review.

Where I draw the line

In my workflow, the agent does not make the final decision.

It can propose a spec. It can split the task. It can write tests. It can write code. It can prepare review evidence. It can record what went wrong. But transitions between stages stay under human control. The final transition, merge into main, is always mine.

That sounds like a small detail until you look at what happens without it.

If the agent moves the task forward by itself, it starts optimizing for the wrong thing. It is easier to close the workflow than to admit the spec is weak. Easier to add a workaround than to go back to the plan. Easier to say "tests passed" than to show which tests were relevant.

It is not malicious. It just keeps executing.

I do not want an autonomous agent that keeps going without inspection. I want a tool that does the heavy work while leaving a clear point where a human accepts responsibility.

So the rule is boring and strict: the AI proposes, I merge.

What I mean by spec-driven

For me, spec-driven development is not a slogan and not a folder full of documents.

It means the agent is not allowed to start with code. First we write down the behavior, the invariants, and the acceptance criteria. What must not break. Which data source is the source of truth. How we will know the task is actually done.

Then the task is split into small contracts. Not "implement the feature", but pieces with a clear expected change and a way to check it.

Only after that does the agent write code.

The loop looks like this:

SPEC -> PLAN -> TEST -> CODE -> REVIEW -> LEARN

I do not run this as a ceremony. I run it as a set of checks.

Before risky code, I want a failing test or at least a concrete scenario. After code, I want a separate review role, usually a separate console with clean context. The reviewer needs regression risk, blast radius, and artifacts, not a retelling of the author's intent.

If a check failed, the plan changed mid-run, or I had to intervene manually, something should remain after the run: a test, a rule, a note, or a hook change. Otherwise the next session will probably repeat the same mistake with a different file name.

The scientific part, without pretending this is a lab

I use the word "scientific" carefully here.

I do not mean that every number in this series is a controlled benchmark. It is not. Some of the measurements are small, local, and tied to my own codebase.

I mean something more basic and more useful for engineering:

  • make a claim;
  • tie it to an artifact;
  • run a check;
  • keep the failure record;
  • change the process when the failure repeats.

That is the difference between "the agent seemed better" and "the agent skipped contract checks in 10 out of 15 runs, here are the transcripts."

The mechanisms I will write about in this series did not come from vibes. Most of them started as an idea from a paper or a known engineering practice: hooks instead of one giant prompt, design-by-contract, context offloading, memory between sessions, clean retry quarantine. Then I tried to make them survive real coding work.

Some survived. Some did not. The ones that did not are usually more useful to talk about.

What the agent is not allowed to do

The first rule is already clear: the agent does not merge its own work.

There are a few more. They are not there because I enjoy process. Each one exists because something broke.

The agent does not review its own code as the final authority. Authors are bad at seeing their own gaps; models are not magically exempt from that.

Production secrets are not a moral slogan. Sometimes an agent can technically read a file. The boundary is where values can end up. They must not be copied into the answer, transcript, log, markdown artifact, diff, or anywhere else that survives the session.

Green CI is mandatory. If CI is red, I do not care that the agent says the change is local.

Invalid output hard-stops the run. If the monitor sees truncated JSON, a missing artifact, or a broken contract, the workflow stops. Not "we will look later." Stop.

And every serious run leaves artifacts under .map/. These files are not exciting. That is the point. They answer the boring questions later: what was the plan, what changed, what was checked, what failed, what did we learn?

Without artifacts, you are left with memory and a convincing narrative after the fact. Agents are very good at convincing narratives.

This is not heavy by default

I do not want enterprise ceremony for a README typo.

The depth depends on risk. A small change gets a short plan, implementation, check, review, and a lesson only if there is actually something worth saving. Risky code gets the full TDD path and separate roles. Sometimes only one subtask needs TDD.

The common part is the accept gate.

The task should not be "done" only because the chat says so. There should be an artifact and a check that explain why it is done.

Otherwise the end state is just a confident answer from the agent, not a result you can review.

Why I went this far

map-framework grew out of a practical problem: coding agents can produce a diff quickly, but long tasks become hard to control without stages and checks. Especially in large Go code, where a mistake does not always fail immediately.

A recent example: reviewer comments around IngressConfig in a Kubernetes operator. The work was not glamorous. Move fields under settings.{nginx,contour}, update tests and CRD/deepcopy, avoid leaking plan markers into Go code.

It ended up as nine commits.

The final verifier ran gofmt, go vet, go build, functional tests, checked CRD invariants, and found one non-blocking issue: an old test comment still said spec.nginx.enabled, while the code already used Spec.Settings.Nginx.Enabled.

That is exactly the kind of thing I can miss in a manual review. In a findings artifact, it is written down. Without that artifact, the agent's output is basically "done, trust me."

After moving to this workflow on a production Go platform with a codebase over a million lines, the monthly amount of code that survived review and reached main grew by roughly 1.8x over six months.

I counted roughly, by lines. It is not a perfect productivity metric. Lines of code are not value. But it was enough signal for me to stop treating the workflow as a personal habit and start treating it as an engineering mechanism.

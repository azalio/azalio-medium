Как написать MCP-сервер, который не разваливается под агентом

MCP-сервер - это не еще один API, а контракт между детерминированным кодом и недетерминированным агентом. 9 практик на примере моего llm-council.

TL;DR

  • Гайд основан на 3 статьях: академическая таксономия паттернов MCP-серверов (arXiv:2606.30317), инженерный разбор от Anthropic и плейбук Block с 60+ серверов - и моем опыте построения mcp, который любят llm-агенты.
  • mcp-server это не еще один API, это контракт между детерминированным кодом и недетерминированным поведением агента.
  • 9 практик для написания действительно хорошего MCP.
  • На каждом шаге пример из моего консула - код открыт.

Сейчас MCP пишут постоянно, берешь fast-mcp, навешиваешь setRequestHandler, зашиваешь какую-то бизнес-логику и все, mcp готов.

Но есть небольшая проблема. Работа на демо != работает устойчиво в проде.

Я использую llm-council каждый день помногу раз и, конечно, постоянно исправлял места где меня жало.

Внизу последовательность шагов, которые помогут сразу написать хороший mcp с примерами и обоснованиями.

Шаг 0: Пойми что ты пишешь не API

Ты не пишешь getWeather("MSK"), ты пишешь эндпоинт для LLM-агента, который на один и тот же запрос может

  • вызвать твой tool
  • придумать свой tool
  • вызвать правильный, но с несуществующими параметрами
  • ответить из головы
  • переспросить
  • ...

Поэтому мы пишем так, чтобы модель из названия и описания тулов могла сделать правильный выбор.

Шаг 1: Выбери одну из 5 форм, до того как начал что-то писать

Академическая статья предложила 5 паттернов проектирования MCP серверов:

  • Resource Gateway (предок: Repository / REST) - отдавать данные бэкенда на чтение
  • Tool Orchestrator (Facade / Mediator) - спрятать мультисистемный workflow за одним вызовом
  • Stateful Session Server (Session / Memento) - ход реально зависит от прошлого состояния, но состояние не видно в промпте
  • Proxy Aggregator (Proxy / API gateway) - выставить много upstream-серверов как один
  • Domain-Specific Adapter (Adapter из GoF) - перевести LLM-недружелюбный API в пригодный

Смысл понятное дело не в том, чтобы навесить ярлык, смысл в том, что паттерн определяет реализацию.

llm-council - это Tool Orchestrator, основной тул ask_council принимает один вопрос, за ним несколько моделей отвечают независимо, вслепую ранжируют ответы друг друга, а модель-председатель выдает итог. Вызывающий этот веер не видит вообще - в этом и суть Tool Orchestrator.

Шаг 2: Режь. Режь по воркфлоу, а не по эндпоинтам

Тут 2 антипаттерна:

  • God tool - один tool на все
  • Тонкая обертка - десятки tools на все твои эндпоинты типа get_user, get_file, list_events.

Правильная единица не эндпоинт, не все сразу, а воркфлоу.

У llm-council ask_council - это один воркфлоу (весь совет: ответы, ранжирование, синтез) за одним вызовом. Не God Tool - потому что у него одна ясная задача, а не строковый action. Не обёртка - потому что за ним не один эндпоинт, а вся многомодельная механика.

Шаг 3: Долгие операции - через async

Практика проста: если твой mcp работает дольше десятка секунд - делай асинхронный эндпоинт.

У llm-council в deep режиме обсуждение может идти до 40 минут. Некоторые кодинг-агенты умеют так долго ждать, а некоторые напрочь отказываются. Это меня жало больше всего, и я перешел на async/poll достаточно давно.

Сразу возвращаем job-id и даем инструкцию агенту, чтобы он поллил по этому id статус запроса: start_council_async отдает task_id, дальше poll_council_task раз в 15-30 секунд, пока статус не станет done или error.

Шаг 4: Описание тула - это промпт, а не комментарий

Модель выбирает инструмент по описанию. Не по схеме, не по отдельной доке - по описанию.

Anthropic формулирует так: пиши описание как объяснение новому сотруднику, который не может переспросить - потому что модель это ровно он и есть. Отсюда:

  • не user, а user_id
  • может быть enum - делай enum, а не строку
  • весь протокол использования - в описание, а не в README, которое модель не увидит

У llm-council описания тулов начинаются не с «что я делаю», а с «когда меня звать». ask_council: «если твой клиент не может ждать дольше пары минут - иди в start_council_async». get_available_models: «позови до совета, если нужно сказать пользователю, какие модели будут отвечать». list_conversations: «потерял conversation_id - тут найдешь». Модель выбирает из восьми тулов, и критерий выбора ей подсказывает само описание, а не метод тыка.

Насколько это важно? После точечной правки описаний внутренних тулов Sonnet выбил SOTA на SWE-bench. Разница между «работает» и «работает надежно» часто именно в тексте описания.

Шаг 5: Возвращай сигнал, а не сырые данные

Контекстное окно у модели ограничено. Ответ твоего mcp ложится в это же окно - и место, которое занял мусор из твоего JSON, уже не достанется самой задаче.

  • Возвращай поля, которые двигают следующее действие: name, file_type. UUID и mime_type - мусор в контексте.
  • У тула, который может вернуть много, проверяй размер до отдачи. В файловом туле Goose файлы больше 400 КБ роняют вызов с ошибкой - модель достаточно умна, чтобы обойти через sed -n '500,1000p'.
  • Ошибка - тоже промпт. Плохо: ERR_TOO_LARGE. Хорошо: «файл 800 КБ, лимит 400, возьми кусок через head/tail/sed».

У llm-council: get_council_metrics отдает точные KPI (успехи, деградации, hit-rate кэша, латентность по стадиям), а не сырой лог прогонов. get_council_eta говорит, сколько ждать, ДО того как агент решится запускать долгий прогон.

И оба честно говорят, каким цифрам можно верить: пока сэмплов мало, ETA возвращает null с пометкой insufficient_data, а не выдуманное число. Модель не может проверить твою методологию - она поверит любой цифре, которую ты ей дашь.

Шаг 6: Сделай состояние явным

В статье "MCP Server Architecture Patterns for LLM-Integrated Applications" есть собственное измерение: два LLM-оценщика классифицировали 54 сервера по описаниям инструментов. И нашли слепое пятно: сессионное состояние в описаниях тулов не видно вообще. Любой stateful-сервер снаружи читается как stateless.

У меня это слепое пятно было настоящим. ask_council принимает опциональный conversation_id - передашь, и совет учитывает историю разговора. Но опиши сервер одной строкой - «спрашивает несколько моделей и синтезирует ответ» - и про сессию никто не догадается.

Хуже того, отказ тихий: не передал id на follow-up - сервер не упадет, а молча заведет новый разговор без памяти о старом. Отказ без ошибки - худший вид отказа: его не ловит ни ретрай, ни обработчик исключений.

Поправил, конечно, теперь так:

  • назван сам факт: «IMPORTANT - this tool is stateful across calls», скопируй id и передай на следующем вызове
  • назван тихий отказ: не вернешь id - будет новый разговор, без ошибки
  • дан путь восстановления: poll_council_task возвращает conversation_id отдельным полем еще до done, а list_conversations описан как спасательный круг - потерял id, позови меня, нужный обычно первый сверху

Шаг 7: Чисти все, что течет в контекст

Все, что твой сервер кладет в контекст модели, - это вход, которому модель поверит. Файл, строка из базы, текст ошибки.

  • У id известная форма - валидируй форму. URL должен вести на известные хосты - заведи allow-list.
  • Auth по-взрослому: OAuth, минимальные scope, токены в keyring, не в плейнтексте.
  • Аннотации типа readOnlyHint - клиенты используют их для умного подтверждения вызовов. Не мешай read и write в одном инструменте.

llm-council локальный и наружу базы не отдает, но один канал утечки есть и у него: тексты ошибок. Ошибка апстрима легко приносит в себе заголовок авторизации - и уезжает прямиком в контекст, а оттуда в лог транскрипта. Поэтому каждая ошибка проходит через sanitize_error: Bearer ..., sk-..., api_key=... режутся в [REDACTED].

Шаг 8: Бюджет всему голова

Измеренное число из статьи: точность выбора tool падает ниже 90% после 10-15 инструментов в контексте для Haiku 4.5 и после 20-30 для Sonnet 4.

У llm-council 8 тулов - в безопасном диапазоне даже под жесткий Haiku-бюджет.

Прием, который помогает под потолком держаться: новая возможность != новый тул. Когда у совета появились режимы глубины (quick/standard/deep), это стало параметром mode у существующих инструментов, а не тремя новыми.

Шаг 9: Тестируй на прогонах, а не на глаз

Ты не можешь на глаз сказать, удобны ли твои тулы агенту. Надо мерить.

Полный цикл от Anthropic: сгенерируй десятки задач на реальных сценариях, прогони агента по каждой, собери точность, число вызовов и токены, потом скорми транскрипты обратно модели - она отлично находит противоречивые описания и кривые схемы. У них Claude Code так сам оптимизировал внутренние тулы и обгонял ручные «экспертные» реализации.

У llm-council для этого в репе лежит бенчмарк: 12 промптов (логика, код, математика) с эталонными ответами. По нему я выбираю состав совета: каждая модель-кандидат отвечает на все промпты, потом все вслепую ранжируют ответы друг друга (голос за свой ответ выкидывается), а нейтральный судья вне пула оценивает итог. Кто в совете, а кто председатель - решает композит из этих цифр, а не мое «эта модель вроде умная».

Поверхность тулов сторожит отдельный скрипт audit_mcp_surface.py: сверяет живые описания и схемы с golden-снапшотом и валит проверку, если что-то поехало - переименовался аргумент, утек внутренний параметр, вылез лишний тул. Антипаттерны из той самой статьи ловятся диффом, а не глазами.

Минимальный вариант, если ничего этого нет, - тест на салфетке: дай модели список своих тулов и десять реальных запросов, попроси выбрать тул на каждый. Точность ниже 90% - проблема не в модели, а в твоих описаниях.

Где гайд заканчивается

Две вещи в llm-council не укладываются ни в один из 5 паттернов. Думаю, это не пробел моего сервера, а пробел карты: исследование строили на серверах, у которых бэкенд - базы и API, а не сами LLM.

Семантический кэш ответов. Прогон совета дорогой, повторный вопрос жалко гонять заново. Но отдать не тот кэшированный ответ - хуже, чем потратить деньги. Поэтому три сита по убыванию доверия: почти дословный повтор (перекрытие токенов ≥ 0.9) - отдаем из кэша сразу; близкий парафраз по локальному эмбеддингу (≥ 0.86) - тоже; серая зона (0.68-0.86) - валидация председателем: «можно ли этим кэшированным ответом безопасно ответить на новый вопрос, да/нет».

Адаптивная маршрутизация моделей. Не всякому вопросу нужен полный совет. Но решать «нужен или нет» еще одной LLM - значит платить за экономию. Поэтому детерминированные правила: рутинный вопрос (auto-режим, короче 120 слов, без маркеров риска типа production/security/migration) стартует тройкой моделей по приоритету, а не полным пулом. Дальше эскалация по триггерам: упала одна из моделей, ответов или ранжирований меньше двух, уверенность совета низкая - маршрут раскрывается до полного пула, и в метаданных остается след, почему именно.

Гайд коротко

  1. Ты пишешь не API, а контракт с недетерминированным читателем.
  2. Выбери одну из 5 форм до первого тула.
  3. Режь по воркфлоу, не по эндпоинтам и не в God Tool.
  4. Долгие операции - job-id + poll, с первого дня.
  5. Описание инструмента - промпт для новичка, который не переспросит.
  6. Возвращай сигнал, не сырые данные. Ошибки - actionable.
  7. Состояние - явно, в описаниях тулов, с путем восстановления.
  8. Чисти все, что течет в контекст. Auth и уровни риска - до прода.
  9. Меньше 10-15 тулов в контексте.
  10. Тестируй на данных, минимум - тест на салфетке из десяти запросов.
